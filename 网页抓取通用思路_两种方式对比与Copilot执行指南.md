# 网页抓取通用思路：两种方式对比与 Copilot 执行指南

## 1. 文档目的
这份文档回答三个问题：
- 你当前两个 Python 脚本各自采用了什么抓取方式，差异在哪里。
- 做网页数据抓取时，通用的思路和原理是什么。
- 在 VS Code + GitHub Copilot 中，完成此类任务可以走哪些执行路径（逆向 API、MCP、浏览器自动化等）。

## 2. 两个脚本的方式对比
对比文件：
- `michelin_scrape.py`
- `deprecated/michelin_scrape_deprecated.py`

### 2.1 方式 A：接口直连（当前脚本）
核心机制：
- 直接调用站点前端背后的 Algolia 接口。
- 通过 `filters` + 分页拿到结构化 JSON。
- 一次性获取中国数据，再按城市分文件输出。

关键特征：
- 库依赖：`requests`。
- 数据来源：API 响应的 `hits`。
- 分页方式：`page` + `nbPages`。
- 数据完整度：高（取决于接口返回字段）。
- 抓取效率：高（请求量少，吞吐大）。

### 2.2 方式 B：浏览器自动化（废弃脚本）
核心机制：
- 使用 Playwright 打开页面、滚动、找链接、翻页。
- 再逐个打开餐厅详情页，从 DOM / JSON-LD /正文文本中抽字段。

关键特征：
- 库依赖：`playwright`。
- 数据来源：页面元素与页面内嵌结构化数据。
- 分页方式：抓取列表页里的 next/page 链接。
- 数据完整度：中等到高（受页面加载、选择器稳定性影响）。
- 抓取效率：中等到低（大量页面访问）。

### 2.3 直接结论
- 对米其林这类“前端消费搜索接口”的站点，接口直连通常优于浏览器自动化。
- 浏览器自动化仍然是必要兜底方案，特别是在接口不可复现、依赖复杂交互、或需要登录态时。

## 3. 两种方式的优劣势矩阵

### 3.1 接口直连（Reverse API）
优势：
- 速度快，请求数少。
- 字段结构化，清洗成本低。
- 分页稳定，可全量拉取。
- 代码更短，后续维护成本低。

劣势：
- 需要先定位接口与参数。
- 接口字段、索引、鉴权参数可能变动。
- 部分业务字段可能不在接口返回里。

适用场景：
- 目标站点明显是前端渲染。
- Network 中存在可复现 JSON 接口。
- 任务是全量拉取、批量化更新。

### 3.2 浏览器自动化（Playwright）
优势：
- 能模拟真实用户行为（点击、滚动、登录）。
- 对“无公开稳定接口”的页面仍可抓。
- 可直接抓动态渲染后的内容。

劣势：
- 慢，资源占用高。
- 对选择器和页面结构敏感。
- 容易受加载时序、反爬机制影响。

适用场景：
- 必须交互后才出现数据。
- 接口签名复杂、短期无法复现。
- 需要抓接口之外的视觉文本内容。

## 4. 网页抓取通用思路与原理（可迁移）

### 4.1 总原则：先找数据源，再选抓取手段
优先级建议：
1. 先看有没有可复现的 JSON 接口。
2. 若无，再看页面中内嵌数据（JSON-LD、window.__INITIAL_STATE__）。
3. 最后才使用浏览器自动化从 DOM 抽取。

### 4.2 通用流程（SOP）
1. 明确目标数据模型
- 先定义你要哪些字段（主键、名称、地址、分类、评分、来源 URL 等）。

2. 做网络探测
- 打开 F12 -> Network -> 过滤 Fetch/XHR -> 刷新页面。
- 找到返回目标数据的请求。

3. 最小复现请求
- 固定 URL、Method、Headers、Payload。
- 用脚本先请求 1 页，确认能返回预期字段。

4. 构建分页与过滤
- 确认 `page/offset/limit` 规则。
- 通过 `filters/facets` 做范围收敛（国家、城市、状态等）。

5. 数据清洗与标准化
- 统一空白、时区、货币、地址格式。
- 建立稳定主键（objectID/identifier/url）。

6. 输出与索引
- 按业务维度拆文件（例如按城市）。
- 生成总索引文件，保存计数、文件映射、抓取时间。

7. 质量校验
- 总数校验、去重校验、抽样校验、重跑一致性校验。

### 4.3 关键原理
- 页面不等于数据源：现代站点常由前端 API 驱动，HTML 只是壳。
- 分页是全量抓取的核心：不做分页就只能拿首屏。
- 过滤条件依赖可过滤字段：若 `facets` 暴露了某字段，通常可以用它过滤。
- 字段可用性靠实测：字段在请求里声明，不代表一定有值；必须看响应。

## 5. VS Code + GitHub Copilot 执行这个任务的方式

## 5.1 方式一：代码库内执行（本地脚本）
做法：
- 让 Copilot 直接改 `michelin_scrape.py`。
- 在 VS Code 终端运行并验证输出。

优点：
- 适合本地调试、快速迭代。
- 修改和结果都在当前仓库，便于追踪。

## 5.2 方式二：逆向 API 路径（推荐）
做法：
- 先抓包拿到真实接口。
- Copilot 负责生成可复现请求、分页、清洗、落盘逻辑。

优点：
- 最符合“高效全量抓取”的工程目标。
- 最容易做稳定任务化（定时跑、增量跑）。

## 5.3 方式三：浏览器自动化路径（Playwright）
做法：
- Copilot 写 Playwright 脚本模拟人类浏览。
- 解析 DOM 或 JSON-LD 提取字段。

优点：
- 接口不可用时的强兜底。

注意：
- 应控制频率、等待策略、错误重试和反爬风险。

## 5.4 方式四：MCP 路径（辅助能力）
说明：
- MCP 不是单一抓取算法，而是“能力接入层”。
- 在此任务中可用于：
  - 读取最新文档与 API 参考。
  - 通过 GitHub MCP 直接提交代码到远程仓库。
  - 进行跨工具协同（代码、文档、提交流程）。

你这次已经实际用了一个 MCP 能力：
- 使用 GitHub MCP 直接把文档提交到远程仓库。

## 5.5 方式五：混合策略（生产实践常用）
推荐组合：
- 主链路：逆向 API 全量抓取。
- 兜底链路：Playwright 补抓接口缺失字段。
- 工程链路：Copilot 负责代码生成与重构，MCP 负责文档/提交自动化。

## 6. 如何选择方案（决策树）
1. Network 中能看到稳定 JSON 接口吗？
- 能：优先 Reverse API。
- 不能：进入 2。

2. 页面是否需要复杂交互/登录后才有数据？
- 是：优先 Playwright。
- 否：继续尝试内嵌 JSON 或静态接口。

3. 是否要求高吞吐、低成本、可持续维护？
- 是：尽量回到 API 路径。
- 否：一次性任务可用 Playwright 快速交付。

## 7. 本项目的最佳实践结论
- 当前项目应以 `michelin_scrape.py` 的接口直连模式为主。
- `deprecated/michelin_scrape_deprecated.py` 保留为兜底参考实现。
- 后续若字段需求增加，可先尝试扩展 `attributesToRetrieve`；仍缺失再补 Playwright 二次抓取。

## 8. 后续可扩展建议
- 增加重试、指数退避、请求节流。
- 增加增量更新模式（按更新时间或对象主键比对）。
- 增加 CSV/Parquet 导出，方便分析。
- 增加单元测试与抽样校验脚本。
